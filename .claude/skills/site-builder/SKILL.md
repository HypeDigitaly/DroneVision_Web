# Site Builder Skill

Build production-ready client websites from `blueprint.json` (generated by `/website-planner` skill). This skill automates the complete pipeline from repository creation through Netlify deployment, with a mandatory human review checkpoint before live deployment.

## Stack & Technology

- **Astro 5.16** — Static site generation with minimal JavaScript
- **Tailwind CSS 4.1** — Utility-first styling with CSS variables for branding
- **TypeScript** — Type-safe components and scripts
- **Netlify** — Hosting (Functions + Blobs for contact forms & lead magnet storage)
- **Resend** — Transactional email API for form submissions
- **GitHub** — Repository hosting and template cloning
- **Language Support** — Bilingual (Czech `_cs` / English `_en`) inline keys in `site.json`

## Prerequisites

Before running this skill, ensure:

1. **CLI Tools Installed & Authenticated**
   - `gh` CLI authenticated (`gh auth status` returns authenticated)
   - `netlify` CLI authenticated (`netlify status` returns account info)
   - Node.js 20+ installed (`node --version`)
   - Git configured (`git config user.email` returns email)

2. **GitHub Template Repository**
   - `HypeDigitaly/hype-client-template` exists as a GitHub Template Repository
   - Verified accessible with authenticated `gh` CLI

3. **Blueprint Approved**
   - `blueprint.json` file exists locally
   - Schema validated (see **Validation Rules** below)
   - Client has approved content, branding, and configuration

4. **Assets Available**
   - Logo file (PNG/SVG) referenced in `blueprint.deployment.assets.logo`
   - Hero images, case study images referenced and available
   - Lead magnet PDF (if enabled) available

## Input: blueprint.json Schema

**SECURITY: blueprint.json must NEVER contain actual secrets.** All sensitive values use `$VAR` references resolved at deploy time from the operator's environment variables.

The `blueprint.json` file (output from `/website-planner`) contains:

```json
{
  "clientSlug": "prodej-cz",
  "buildConfig": {
    "brandColors": {
      "primary": "#0066cc",
      "secondary": "#ff6b35",
      "accent": "#ffd93d"
    },
    "typography": {
      "fontFamily": "Inter, sans-serif",
      "headingFamily": "Poppins, sans-serif"
    },
    "sections": ["hero", "features", "testimonials", "contact"]
  },
  "siteJson": {
    "meta": {
      "name": "Client Company Name"
    },
    "company": {
      "name": "Client Company",
      "ico": "12345678",
      "email": "contact@client.cz",
      "city": "Praha"
    },
    "contact": {
      "formNotificationEmails": [
        "admin@client.cz"
      ]
    }
  },
  "deployment": {
    "envVars": {
      "RESEND_API_KEY": "$RESEND_API_KEY",
      "NOTIFICATION_EMAILS": "admin@client.cz",
      "ADMIN_PASSWORD_HASH": "$ADMIN_PASSWORD"
    },
    "assets": {
      "logo": "/path/to/logo.png",
      "heroImage": "/path/to/hero.jpg",
      "images": ["/path/to/img1.jpg"]
    }
  }
}
```

> **Note:** `$RESEND_API_KEY` and `$ADMIN_PASSWORD` are resolved from the operator's environment at deploy time -- never stored in files.

## Validation Rules

Before proceeding, validate `blueprint.json`:

```
✓ clientSlug: alphanumeric + hyphens, 3-50 chars
✓ siteJson.meta.name: not empty
✓ siteJson.company.name: not empty
✓ siteJson.company.ico: 8 digits
✓ siteJson.company.email: valid email format
✓ siteJson.contact.formNotificationEmails[0]: valid email
✓ buildConfig.brandColors: valid hex colors (regex: /^#[0-9a-fA-F]{6}$/)
✓ deployment.envVars.RESEND_API_KEY: is "$RESEND_API_KEY" (env ref, not actual key)
✓ deployment.envVars.ADMIN_PASSWORD_HASH: is "$ADMIN_PASSWORD" (env ref, not actual hash)
✓ deployment.assets.logo: file exists and <5MB
```

> **Security check:** If any `deployment.envVars` value does NOT start with `$`, reject the blueprint -- actual secrets must never be stored in blueprint.json.

If validation fails, return error summary and stop.

## Sequential Runbook: 8 Steps

### Step 1: Pre-flight Checks

**Objective:** Verify all tools, environment, and inputs before repository creation.

**Actions:**

1. **Verify CLI tools are available and authenticated:**
   ```bash
   gh auth status
   netlify status
   node --version
   git config user.email
   ```
   - If any tool missing or not authenticated, stop with clear error
   - Report versions for troubleshooting

2. **Validate `blueprint.json`:**
   - Check file exists and is valid JSON
   - Verify all required fields present (see schema above)
   - Validate `clientSlug` format (alphanumeric + hyphens, 3-50 chars)
   - Validate email formats in `siteJson.company.email` and `contact.formNotificationEmails`
   - Validate `buildConfig.brandColors` are valid hex colors
   - If validation fails, output detailed error report

3. **Extract clientSlug:**
   ```bash
   CLIENT_SLUG=$(jq -r '.clientSlug' blueprint.json)
   REPO_NAME="${CLIENT_SLUG}_Web"
   ```
   - Report: `Client slug: {CLIENT_SLUG}`, `Repository will be: HypeDigitaly/{REPO_NAME}`

4. **Verify template repository exists:**
   ```bash
   gh repo view HypeDigitaly/hype-client-template --json nameWithOwner
   ```
   - If template not found, stop with: "Template repository HypeDigitaly/hype-client-template not found. Verify it exists and is accessible."

5. **Check for naming conflicts:**
   ```bash
   gh repo view HypeDigitaly/{REPO_NAME} 2>/dev/null && echo "CONFLICT" || echo "AVAILABLE"
   ```
   - If repo exists, ask: "Repository HypeDigitaly/{REPO_NAME} already exists. Proceed to update existing repo? [yes/no]"
   - If no, stop

**Success Output:**
```
✓ Pre-flight checks passed
  - gh CLI: authenticated
  - netlify CLI: authenticated
  - Node.js: v20.x.x
  - blueprint.json: valid
  - Client slug: {CLIENT_SLUG}
  - Repository: HypeDigitaly/{REPO_NAME}
  - Template: accessible
  Ready to proceed with Step 2.
```

**Rollback:** None required (non-destructive checks only).

   ```bash
   git clone https://github.com/HypeDigitaly/{REPO_NAME}.git
   ```

**Success Output:**


### Step 3: Inject site.json

**Objective:** Write site configuration (content, metadata, company info) from `blueprint.siteJson` to `src/data/site.json`.

**Actions:**

1. **Extract siteJson from blueprint:**
   ```bash
   jq '.siteJson' blueprint.json > site_content.json
   ```

2. **Verify all required keys present:**
   - Required keys:
     - `meta.name`
     - `company.name`, `company.ico`, `company.email`, `company.city`
     - `contact.formNotificationEmails` (array with at least 1 email)
   - If missing, report missing keys and stop

3. **Merge with template defaults:**
   - Read current `src/data/site.json` (template version)
   - Deep merge `site_content.json` into template
   - Preserve template structure (don't replace entire file)
   - This ensures new template fields don't break builds

4. **Write merged config:**
   ```bash
   # Merge and write (using jq or similar)
   jq -s '.[0] * .[1]' src/data/site.json site_content.json > src/data/site.json.tmp
   mv src/data/site.json.tmp src/data/site.json
   ```

5. **Validate written file:**
   ```bash
   jq . src/data/site.json > /dev/null && echo "Valid JSON" || echo "INVALID"
   ```
   - If validation fails, restore template version and report error

**Success Output:**
```
✓ site.json injected
  - Keys: {count} configuration entries
  - Languages: Czech (_cs), English (_en)
  - Company: {company.name}
  - Contact emails: {count} recipients
  Ready to proceed with Step 4.
```

**Rollback (if failure):**
```bash
git checkout src/data/site.json
```

### Step 4: Copy Assets

**Objective:** Copy logo, images, and documents from `blueprint.deployment.assets` to `public/` directories.

**Actions:**

1. **Create asset directories:**
   ```bash
   mkdir -p public/images
   mkdir -p public/downloads
   mkdir -p public/images/case-studies
   ```

2. **Copy logo:**
   - Source: `blueprint.deployment.assets.logo`
   - Destination: `public/images/logo.png` (or .svg)
   ```bash
   cp {blueprint.logo} public/images/logo.$(file extension)
   ```
   - Validate file size: `< 500KB`
   - If too large, warn: "Logo is {size}MB. Consider compressing to <500KB for performance."

3. **Copy hero and page images:**
   - Source: `blueprint.deployment.assets.images[]`
   - Destination: `public/images/{filename}`
   ```bash
   for image in {image_list}; do
     cp "$image" public/images/
     SIZE=$(du -h "$image" | cut -f1)
     BYTES=$(wc -c < "$image")
     if [ "$BYTES" -gt 5242880 ]; then
       echo "WARN: $image is ${SIZE} (>5MB). Consider optimizing."
     fi
   done
   ```
   > **Note:** Uses `wc -c` for cross-platform byte count (works on Linux, macOS, and WSL). Avoid `stat -f%z` (macOS only) or `stat -c%s` (Linux only).
   - Warn if any image >5MB

4. **Copy hero images with optimization hints:**
   - Check for hero images referenced in `siteJson` sections
   - Copy to `public/images/`
   - If format is not WebP, suggest conversion: "Consider converting {filename} to WebP for 25-35% smaller file size"

5. **Copy downloadables (lead magnet, PDFs):**
   - Source: `blueprint.deployment.assets.downloads[]`
   - Destination: `public/downloads/{filename}`
   ```bash
   for file in {downloads}; do
     cp "$file" public/downloads/
   done
   ```

6. **Verify all assets copied:**
   ```bash
   ls -lh public/images/ public/downloads/
   ```
   - Report: "{count} images copied, {count} downloads copied"

**Success Output:**
```
✓ Assets copied to public/
  - Logo: public/images/logo.{ext} ({size})
  - Images: {count} files copied ({total_size})
  - Downloads: {count} files copied
  ⚠️  Optimization hints (if any >5MB)
  Ready to proceed with Step 5.
```

**Rollback (if failure):**
```bash
rm -rf public/images/* public/downloads/*
git checkout public/
```

### Step 5: Customize Brand Tokens

**Objective:** Apply client branding (colors, typography) to Tailwind configuration and CSS variables.

**Actions:**

1. **Extract brand configuration:**
   ```bash
   PRIMARY=$(jq -r '.buildConfig.brandColors.primary' blueprint.json)
   SECONDARY=$(jq -r '.buildConfig.brandColors.secondary' blueprint.json)
   ACCENT=$(jq -r '.buildConfig.brandColors.accent' blueprint.json)
   FONT_FAMILY=$(jq -r '.buildConfig.typography.fontFamily' blueprint.json)
   HEADING_FAMILY=$(jq -r '.buildConfig.typography.headingFamily' blueprint.json)
   ```

2. **Validate extracted colors are valid hex:**
   ```bash
   for COLOR in "$PRIMARY" "$SECONDARY" "$ACCENT"; do
     if ! echo "$COLOR" | grep -qE '^#[0-9a-fA-F]{6}$'; then
       echo "ERROR: Invalid hex color: $COLOR" && exit 1
     fi
   done
   ```

3. **Update Tailwind CSS configuration:**
   - Edit `tailwind.config.mjs` (ESM format -- project uses Tailwind CSS 4.1 which is ESM-first)
   - Add custom colors to theme:
   ```javascript
   theme: {
     extend: {
       colors: {
         primary: {
           50: '{hex_variant_lighter}',
           500: '{PRIMARY}',
           900: '{hex_variant_darker}'
         },
         secondary: '{SECONDARY}',
         accent: '{ACCENT}'
       },
       fontFamily: {
         sans: ['{FONT_FAMILY}'],
         serif: ['{HEADING_FAMILY}']
       }
     }
   }
   ```
   - Generate color variants using Tailwind color scale or manual definition

4. **Update CSS variables (optional, if using `:root` approach):**
   - Edit `src/styles/tokens.css` (if exists)
   - Add or update:
   ```css
   :root {
     --color-primary: {PRIMARY};
     --color-secondary: {SECONDARY};
     --color-accent: {ACCENT};
     --font-family: {FONT_FAMILY};
     --font-family-heading: {HEADING_FAMILY};
   }
   ```

5. **Verify configuration:**
   ```bash
   # Check for syntax errors
   npm run check 2>&1 | grep -i "tailwind\|config"
   ```
   - If the config file write fails, restore the original and report the error:
   ```bash
   git checkout tailwind.config.mjs src/styles/tokens.css
   echo "ERROR: Brand token application failed. Restored original config."
   ```

**Success Output:**
```
✓ Brand tokens applied
  - Primary color: {PRIMARY}
  - Secondary color: {SECONDARY}
  - Accent color: {ACCENT}
  - Font family: {FONT_FAMILY}
  - Heading family: {HEADING_FAMILY}
  Ready to proceed with Step 6.
```

**Rollback (if failure):**
```bash
git checkout tailwind.config.mjs src/styles/tokens.css
```

### Step 8: Deploy to Netlify

**Objective:** Create Netlify site, configure environment variables, push code, and verify live deployment.

**Actions:**

1. **Ensure development server is stopped:**
   - Kill any running `npm run dev` process
   - Verify: `lsof -i :4321` should return empty

2. **Ensure .gitignore is in place:**
   The repository must contain a `.gitignore` with at minimum:
   ```
   .env
   .env.*
   .env.deploy
   blueprint.json
   *.key
   *.pem
   node_modules/
   dist/
   ```
   - If `.gitignore` is missing or incomplete, create/update it before committing.

3. **Commit current state to Git:**
   **Never use `git add -A` -- it may stage .env files or blueprint.json containing secrets.**
   ```bash
   git add src/ public/ package.json astro.config.mjs tsconfig.json netlify.toml tailwind.config.mjs .gitignore
   git commit -m "Initial client website from blueprint: {company.name}"
   ```
   - Report: "Committed: {number of files} files changed"

4. **Create Netlify site:**
   ```bash
   netlify sites:create --name {client-slug}
   ```
   - Report: "Creating Netlify site: {client-slug}..."
   - Capture site ID and URL
   - Report: "Netlify site created: https://{client-slug}.netlify.app"

5. **Configure environment variables in Netlify:**
   **Never pass secrets directly on the command line -- they will be visible in shell history.**
   ```bash
   # Write env vars to a temporary gitignored file (never committed)
   cat > .env.deploy <<'ENVEOF'
   RESEND_API_KEY=<value from operator's secure env>
   NOTIFICATION_EMAILS=admin@client.cz
   ADMIN_PASSWORD_HASH=<value from operator's secure env>
   NETLIFY_BLOBS_TOKEN=<value from operator's secure env>
   ENVEOF

   # Import into Netlify (production context only)
   netlify env:import .env.deploy --context production

   # Immediately delete the temporary file
   rm .env.deploy
   ```
   - Resolve `$VAR` references from `blueprint.deployment.envVars` using the operator's actual environment variables at deploy time.
   - **Alternative (most secure):** Set environment variables via the Netlify Dashboard UI instead of CLI.
   - Report: "Environment variables configured: {count} variables set"

6. **Link repository to Netlify:**
   ```bash
   netlify link --dir . --id {NETLIFY_SITE_ID}
   ```
   - This enables auto-deploy on `git push`

7. **Push to GitHub:**
   ```bash
   git push origin main
   ```
   - Report: "Code pushed to GitHub"
   - Netlify should automatically trigger a build on `main` branch

8. **Wait for Netlify deployment:**
   ```bash
   # Check deployment status
   netlify status
   ```
   - Wait up to 5 minutes for build to complete
   - Report build progress if available

9. **Verify deployment:**
   ```bash
   # Test homepage loads
   curl -I https://{client-slug}.netlify.app
   ```
   - Check for HTTP 200 response
   - If not ready, wait and retry

10. **Verify all critical pages load:**
   ```bash
   # Test key pages
   curl -I https://{client-slug}.netlify.app/
   curl -I https://{client-slug}.netlify.app/kontakt
   curl -I https://{client-slug}.netlify.app/privacy-policy
   ```
   - All should return 200 or 301 (redirect)
   - Report: "{count} critical pages verified"

11. **Final deployment summary:**
    ```bash
    netlify status
    ```
    - Report live URL: `https://{client-slug}.netlify.app`

**Success Output:**
```
✓ Deployed to Netlify
  - Site: https://{client-slug}.netlify.app
  - Build status: Success
  - Environment: {count} variables configured
  - Pages verified: / /kontakt /privacy-policy

✓ DEPLOYMENT COMPLETE
  Live URL: https://{client-slug}.netlify.app
```

**Rollback (if failure):**

## Error Handling & Troubleshooting

### Step 1 Failures

**Problem:** `gh auth status` fails
- **Cause:** GitHub CLI not authenticated
- **Solution:** `gh auth login` and reauthenticate

**Problem:** `blueprint.json` validation fails
- **Cause:** Missing or invalid fields
- **Solution:** Return detailed error listing missing/invalid keys. User must fix blueprint and retry.

### Step 2 Failures

**Problem:** Template repository not found
- **Cause:** `HypeDigitaly/hype-client-template` doesn't exist or not accessible
- **Solution:** Verify repo exists and `gh` CLI has access. Create template if needed.

**Problem:** Repository creation fails
- **Cause:** Network issue or permissions error
- **Solution:** Retry `gh repo create`. If persistent, check GitHub status.
- **Rollback:** `gh repo delete HypeDigitaly/{REPO_NAME} --yes`

### Step 3 Failures

**Problem:** `site.json` write fails
- **Cause:** Invalid JSON in siteJson or file permissions issue
- **Solution:** Validate JSON and retry. Check file permissions on `src/data/site.json`
- **Rollback:** `git checkout src/data/site.json`

### Step 4 Failures

**Problem:** Asset files not found
- **Cause:** Path in `blueprint.deployment.assets` incorrect
- **Solution:** Verify file paths exist. Update blueprint and retry.

**Problem:** Image too large (>5MB)
- **Cause:** Unoptimized asset
- **Solution:** Warn user and proceed (or stop for optimization). Consider WebP conversion.

### Step 6 Failures

**Problem:** Build fails with TypeScript errors
- **Cause:** Type mismatches in components or data
- **Solution:** Output error details. User fixes types and retries.
- **Rollback:** Revert changes: `git checkout src/`

**Problem:** Missing image or asset in build
- **Cause:** Path mismatch between `site.json` and actual file
- **Solution:** Verify file exists in `public/`. Update `site.json` and rebuild.

### Step 8 Failures

**Problem:** Netlify deployment fails to start
- **Cause:** Build configuration error
- **Solution:** Check `netlify.toml` or `astro.config.mjs`. Ensure build command is correct.

**Problem:** Environment variables not loaded
- **Cause:** Typo in env var name or not set in Netlify UI
- **Solution:** Verify all vars set: `netlify env:list`. Recheck names from blueprint.

**Problem:** HTTP 500 errors after deployment
- **Cause:** Missing environment variables or Netlify Functions error
- **Solution:** Check Netlify Function logs: `netlify logs --tail`
